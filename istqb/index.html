1장. 테스팅의 기초 180분 · 약 8문제

1.1 테스팅이란 무엇인가

테스팅 = 결함을 식별하고 산출물의 품질을 평가하는 일련의 활동. 대상 산출물 = 테스트 대상(test object).

흔한 오해 2가지 — ① 테스팅 ≠ 테스트 실행 (계획~완료 전체가 테스팅) ② 테스팅 ≠ 베리피케이션만 (밸리데이션도 포함)
  • 베리피케이션: 요구사항대로 만들었나? ("올바르게 만들었나")
  • 밸리데이션: 사용자가 진짜 필요한 걸 만들었나? ("올바른 것을 만들었나")
  • 동적 테스팅: 소프트웨어 실행함 / 정적 테스팅: 실행 안 함(리뷰·정적 분석)

테스팅 vs 디버깅 K2

테스팅은 장애를 유발하거나 결함을 찾는 것. 디버깅은 원인(결함)을 분석·제거하는 것(개발자 일). 디버깅 프로세스: 재현 → 분석 → 수정. 이후 확인 테스팅(+필요시 리그레션 테스팅). 정적 테스팅으로 찾은 결함은 장애가 없으므로 재현·분석 불필요.

1.2 테스팅이 왜 필요한가

테스팅 vs 품질 보증(QA) K1단골

테스팅 (품질 제어)품질 보증(QA)
초점제품 지향프로세스 지향
접근교정(corrective)예방(preventive)
책임테스트 활동프로젝트 참여자 전원

오류 → 결함 → 장애 K2단골

사람의 오류(실수) → 산출물의 결함(버그) → 실행 시 장애 발생 가능
  • 결함이 항상 장애를 일으키는 건 아님 (특정 상황에서만, 또는 전혀)
  • 장애 원인이 결함만 있는 것도 아님 — 환경 조건(방사선, 전자기장)도 가능
  • 근본 원인: 오류를 만든 근본 이유. 제거하면 유사 결함 재발 예방

1.3 테스팅의 7가지 원리 K2필수 암기

  1. 테스팅은 결함의 존재를 밝히지, 없음을 증명하지 못한다
  2. 완벽한 테스팅은 불가능하다 → 기법·우선순위·리스크 기반으로 집중
  3. 조기 테스팅으로 시간·비용 절약 (= 시프트 레프트의 근거)
  4. 결함은 집중된다 — 소수 컴포넌트에 몰림 (파레토 원리)
  5. 같은 테스트 반복 시 효과가 줄어든다 (단, 자동 리그레션은 예외적으로 유익)
  6. 테스팅은 정황(context)에 의존적이다 — 만능 접근법 없음
  7. 결함-부재는 궤변 — 결함이 없어도 사용자 기대에 못 미치면 실패 (밸리데이션 필요)
1번 vs 7번 구별: 1번 = "결함이 없다고 증명 못 함", 7번 = "결함이 없어도 성공 아님"

1.4 테스트 활동, 테스트웨어, 역할

테스트 활동 7가지 K2

계획 → 모니터링·제어 → 분석 → 설계 → 구현 → 실행 → 완료
  • 분석 = "무엇을 테스트할 것인가" → 테스트 컨디션 도출
  • 설계 = "어떻게 테스트할 것인가" → 테스트 케이스 구체화
  • 순차처럼 보이지만 실제로는 반복적·병렬적 수행

활동별 테스트웨어 K2

활동대표 산출물
계획테스트 계획서, 일정, 리스크 관리 대장, 시작/완료 조건
모니터링·제어진행 상황 보고서, 제어 지침
분석테스트 컨디션, 베이시스 결함 보고서
설계테스트 케이스, 테스트 차터, 커버리지 항목, 데이터/환경 요구사항
구현테스트 절차·스크립트·스위트, 테스트 데이터, 실행 일정, 스텁·드라이버
실행테스트 로그, 결함 보고서
완료완료 보고서, 개선 항목, 교훈, 변경 요청

추적성 K2

베이시스 ↔ 테스트웨어 ↔ 결과 ↔ 결함의 연결 유지. 가치: 커버리지 평가, 변경 영향 파악, 감사 지원, 이해하기 쉬운 보고.

역할 2가지 K2

  • 테스트 관리 역할: 프로세스·팀 전반 — 계획, 모니터링, 제어, 완료
  • 테스팅 역할: 공학·기술적 측면 — 분석, 설계, 구현, 실행
  • 한 사람이 겸임 가능

1.5 필수 기술과 모범 사례

테스터에게 필요한 기술 K2

테스팅 지식 · 철저함/호기심/세부 주의력 · 의사소통(경청·팀플레이) · 분석적/비판적 사고와 창의성 · 기술 지식 · 도메인 지식. 테스터는 나쁜 소식을 전하는 사람 — 확증 편향 때문에 결함 보고가 비판으로 오해받기 쉬워 건설적 전달이 핵심.

전체 팀 접근법 K1

XP에서 유래. 전원이 품질에 책임, 기술만 있으면 누구든 어떤 작업이든 수행. 단 안전 치명적 시스템엔 부적합할 수 있음(높은 독립성 필요) ← 함정 선지.

테스팅의 독립성 K2

저자 본인 → 같은 팀 동료 → 팀 외부(조직 내) → 조직 외부 (독립성 낮음 → 높음)
  • 장점: 다른 편향 → 다른 유형의 결함 발견, 가정에 이의 제기
  • 단점: 소통 단절·적대 관계, 개발자의 품질 책임감 상실, 병목 인식
  • 독립성이 친숙함을 대체하진 않음 — 개발자도 자기 코드 결함을 잘 찾음
  • 실무 권장: 여러 수준 혼합 (개발자→컴포넌트, 테스트팀→시스템, 비즈니스→인수)

2장. SDLC와 테스팅 130분 · 약 6문제

2.1 SDLC에서의 테스팅

SDLC 모델 = 개발 프로세스의 추상적 표현. 순차적(폭포수, V-모델) / 반복적(나선형, 프로토타이핑) / 점진적(통합 프로세스). 개발 방법·프랙티스: ATDD, BDD, DDD, XP, FDD, 칸반, 린, 스크럼, TDD.

SDLC가 테스팅에 미치는 영향 K2

모델테스팅 특징
순차적테스터는 초기에 리뷰·분석·설계 참여, 동적 테스팅은 후반에 몰림
반복/점진매 주기 증분 → 모든 레벨 정적+동적 매 주기 가능, 빠른 피드백 + 광범위한 리그레션 필요
애자일변화 상시 가정 → 가벼운 문서 + 자동화 선호, 수동은 경험 기반 기법 위주

좋은 테스팅 프랙티스 4가지 K1

  1. 모든 개발 활동마다 상응하는 테스트 활동
  2. 테스트 레벨마다 구체적·독립적 테스트 목적
  3. 테스트 분석·설계를 상응하는 개발 단계에서 시작
  4. 문서 초안이 나오는 즉시 리뷰 참여

테스트 우선 접근법 K1

접근법구별 키워드
TDD테스트 먼저 → 코드 → 리팩토링 (개발자 중심)
ATDD인수 조건에서 테스트 도출
BDDGiven/When/Then 자연어 → 자동 실행 테스트로 변환

DevOps와 테스팅 K2

  • 이점: 빠른 피드백 / CI가 시프트 레프트 장려 / 안정적 테스트 환경 / 비기능 가시성↑ / 수동 반복 감소 / 자동 리그레션 확대
  • 리스크: 파이프라인 구축 비용, CI/CD 도구 유지보수, 자동화 추가 자원
"DevOps면 수동 테스팅 불필요" = 오답. 사용자 관점 수동 테스팅은 여전히 필요.

시프트 레프트 K2

테스팅을 SDLC 초기로 당기기: 명세 리뷰, 코드 전 테스트 작성, CI/CD, 동적 전 정적 분석, 비기능을 컴포넌트 레벨부터.

① 후반 테스트를 무시해도 된다는 뜻 아님 ② 초기 비용은 오히려 증가, 후반 절감을 기대하는 것

회고 K2

주기 끝·마일스톤에서: ① 무엇이 성공적이었나 ② 무엇이 부족했나 ③ 어떻게 개선·유지할까. 결과는 기록(보통 테스트 완료 보고서에 포함). 이점: 효과성↑, 테스트웨어 품질↑, 팀 학습, 베이시스 품질↑, 협업 개선.

2.2 테스트 레벨과 테스트 유형

레벨 = "언제/어느 단계" (함께 관리되는 활동 집합) / 유형 = "무엇을 확인하나" (품질 특성 관련, 모든 레벨에서 가능)

테스트 레벨 5가지 K2단골

레벨초점특징

테스트 유형 4가지 K2

  • 기능: "무엇을" 하는가 — 완전성·정확성·적합성
  • 비기능: "얼마나 잘" — ISO 25010: 수행 효율성, 호환성, 유용성, 신뢰도, 보안, 유지 가능성, 이동성, 안전성. 늦게 발견하면 심각한 위협 → 조기 시작
  • 블랙박스: 명세 기반 — 구현 바뀌어도 동작 같으면 테스트 유효
  • 화이트박스: 내부 구조 기반 — 구조를 충분히 커버

확인 vs 리그레션 테스팅 K2단골

확인 테스팅리그레션 테스팅

방법실패했던 테스트 재실행 or 새 테스트영향도 분석으로 범위 결정
범위수정 부분다른 컴포넌트·연결 시스템·환경까지

리그레션은 반복 실행·계속 증가 → 자동화 1순위, 프로젝트 초기에 시작.

2.3 유지보수 테스팅 K2

운영 중 시스템 변경 시: ① 변경이 잘 됐는지 + ② 변경 안 된 영역의 리그레션 확인. 범위 결정 3요소: 변경 리스크 / 기존 시스템 크기 / 변경 크기.

계기 3가지

  • 수정사항: 계획된 개선, 결함 수정, 핫픽스
  • 환경 업그레이드/마이그레이션: 새 환경 테스트 + 데이터 변환 테스트
  • 단종: 데이터 보관(archiving) 테스트 + 복원·복구 테스트
"단종에도 테스팅이 필요하다"가 의외의 출제 포인트

3장. 정적 테스팅 80분 · 약 4문제

3.1 정적 테스팅의 기초

정적 테스팅 = 소프트웨어를 실행하지 않고 평가. 두 갈래:

  • 리뷰 — 사람이 수동으로 (요구사항, 코드, 계획서 등)
  • 정적 분석 — 도구가 자동으로 (테스트 케이스 불필요, CI에 통합하는 경우 많음. 맞춤법 검사기도 정적 분석 도구의 예)

검사 가능한 산출물 K1

  • 가능: 요구사항 명세, 소스 코드, 테스트 계획서, 테스트 케이스, 백로그 항목, 계약서, 모델 등 — 읽고 이해할 수 있는 거의 모든 것
  • 정적 분석은 구조(문법이 정해진 코드·모델)가 필요
  • 부적합: 사람이 해석하기 어려운 것, 법적으로 분석 금지된 타사 실행 코드

정적 테스팅의 가치 K2

4.1 테스트 기법 개요 K2

테스트 기법 = 테스트 분석("무엇을")과 설계("어떻게")를 지원. 적은 수의 충분한 테스트 케이스를 체계적으로 만들게 해줌.

분류기반특징
블랙박스 (명세 기반)명시된 동작구현과 무관 → 구현이 바뀌어도 동작이 같으면 테스트 유효
화이트박스 (구조 기반)내부 구조·처리설계/구현이 끝난 후에 작성 가능
경험 기반테스터의 지식·경험효과가 테스터 능력에 좌우. 블랙/화이트가 놓치는 결함 발견 → 보완 관계

4.2 블랙박스 테스트 기법 전부 K3 — 계산 문제!

① 동등 분할 (EP)

같은 방식으로 처리될 값들을 분할로 묶고, 분할당 값 하나만 테스트. 근거: 한 분할의 어떤 값으로 결함을 찾으면 같은 분할의 다른 값으로도 찾을 수 있다.

  • 유효 분할(처리해야 하는 값) / 비유효 분할(무시·거부해야 하는 값) — 비유효 분할도 반드시 커버
  • 분할은 겹치면 안 되고, 공집합이면 안 됨
  • 커버리지 = 실행한 분할 수 ÷ 전체 분할 수 × 100%
  • 분할 집합이 여러 개일 때(입력 변수 2개 이상): 이치 초이스(Each Choice) 커버리지 — 각 분할을 최소 1회씩, 조합은 고려 안 함
예제: 나이 입력 18~65가 유효라면 → 분할 3개: [~17 비유효] [18~65 유효] [66~ 비유효] → 테스트 케이스 3개면 100%

② 경계값 분석 (BVA)

결함은 분할의 경계에 몰린다 (개발자가 <, ≤를 헷갈리니까). 정렬된 분할에만 사용 가능. 분할의 최솟값·최댓값이 경계값.

2-value BVA3-value BVA
커버리지 항목경계값 + 인접 분할의 가장 가까운 값경계값 + 양쪽 이웃 값
엄격함기본더 엄격 (2-value가 못 찾는 결함 발견)
단골 예제: "if (x≤10)"을 "if (x=10)"으로 잘못 구현 → 2-value(10, 11)로는 못 잡고, 3-value의 9가 잡아냄
예제: 18~65 유효일 때 경계 18의 2-value = {17, 18}, 3-value = {17, 18, 19}. 경계 4개(17|18, 65|66) 기준 2-value 총 4개 값, 3-value 총 6개 값

③ 결정 테이블 테스팅

조건 조합에 따라 달라지는 결과(비즈니스 규칙)를 테이블로 정리. 행 = 조건과 동작, 열 = 결정 규칙 하나.

  • 표기: 조건 T(참) / F(거짓) / –(결과에 영향 없음) / N/A(실행 불가), 동작 X(발생) / 공백(발생 안 함)
  • 제한-입력(모두 부울값) vs 확장-입력(숫자 범위·동등 분할 등 복수 값)
  • 조건 n개 → 규칙 최대 2ⁿ개로 폭발 → 실행 불가능한 열 삭제·병합으로 최소화, 또는 리스크 기반 접근
  • 커버리지 = 실행한 열 ÷ 실현 가능한 열 × 100%
  • 강점: 모든 조건 조합을 체계적으로 식별, 누락·모순된 요구사항 발견

④ 상태 전이 테스팅

상태 다이어그램으로 시스템 동작 모델링. 전이 표기: "이벤트 [가드 조건] / 동작"

  • 상태 테이블: 행=상태, 열=이벤트, 셀=전이. 다이어그램과 달리 비유효 전이를 빈 셀로 명확히 보여줌
  • 커버리지 3종 (약함 → 강함):
    1. 모든 상태 커버리지 — 모든 상태를 1회 이상 (가장 약함)
    2. 유효 전이 커버리지 (0-스위치) — 모든 유효 전이 실행. 가장 널리 사용. 100% 달성 시 모든 상태도 달성
    3. 모든 전이 커버리지 — 유효 전이 + 비유효 전이 시도까지. 100% 달성 시 위 둘 다 보장. 미션·안전 치명적 SW의 최소 기준
  • 비유효 전이는 테스트 케이스당 하나만 — 결함 마스킹(한 결함이 다른 결함 탐지를 방해) 방지

4.3 화이트박스 테스트 기법 K2 — 설명만

구문 테스팅 / 구문 커버리지

커버리지 항목 = 실행 가능한 구문. 커버리지 = 실행한 구문 ÷ 전체 실행 가능 구문 × 100%.

  • 100%여도 결함을 다 찾는 건 아님 — 데이터 종속 결함(예: 분모가 0일 때만 실패하는 나눗셈)은 못 잡음
  • 100%여도 모든 분기를 테스트한 건 아님

분기 테스팅 / 분기 커버리지

분기 = 제어 흐름 그래프에서 두 노드 간 제어 이동 (무조건 / 조건: if의 참·거짓, switch, 루프 계속·종료).

100% 분기 커버리지 ⊃ 100% 구문 커버리지 (역은 성립 안 함) ← 단골 문제

화이트박스의 가치

  • 강점: 구현 전체를 고려 → 명세가 모호·불완전해도 결함 감지. 객관적인 커버리지 측정치 제공 (블랙박스만으론 코드 커버리지 측정 불가)
  • 약점: 요구사항이 구현 안 된 경우(누락)는 발견 못 함
  • 정적 테스팅(코드 드라이 런)이나 슈도코드 검토에도 사용 가능

4.4 경험 기반 테스트 기법 K2

오류 추정

테스터의 지식으로 오류·결함·장애를 예측: ① 앱의 과거 동작 ② 개발자가 잘 저지르는 오류 유형 ③ 유사 앱의 장애 유형. 구현 방법 = 결함 공격(발생 가능한 오류·결함 목록을 만들어 그걸 노리는 테스트 설계).

탐색적 테스팅

  • 테스트의 설계·실행·평가를 동시에 하면서 대상을 배워감
  • 세션 기반: 시간을 정하고(타임박스), 테스트 차터(목적)를 지침으로, 끝나면 보고·토론
  • 명세가 부족하거나 시간 압박이 심할 때 유용, 다른 공식 기법 보완
  • 테스터의 경험·도메인 지식·분석 기술이 좋을수록 효과적

체크리스트 기반 테스팅

  • 체크리스트 항목 = 보통 질문 형식, 개별적·직접적으로 확인 가능해야
  • 포함 금지: 자동 점검 가능한 항목, 시작/종료 조건이 더 적합한 항목, 너무 일반적인 항목
  • 시간이 지나면 효과 감소 → 결함 분석 기반으로 정기 업데이트 (단, 너무 길어지지 않게)
  • 커버리지는 높아지지만 재현 가능성은 떨어질 수 있음

4.5 협업 기반 테스트 접근법

4.2~4.4가 결함 식별이 목표라면, 협업 기반은 결함 예방에도 초점.

사용자 스토리 작성 K2

  • 3C: 카드(Card, 스토리 매체) / 대화(Conversation, 사용 방법 설명) / 확인(Confirmation, 인수 조건)
  • 형식: "[역할]로서 [목표]를 달성해 [비즈니스 가치]를 얻기를 원한다"
  • 좋은 스토리 = INVEST: Independent, Negotiable, Valuable, Estimable, Small, Testable
  • 테스트 방법을 모르겠으면 → 스토리가 불명확하거나 중요한 내용이 빠진 신호

인수 조건 K2

스토리 승인을 위해 충족해야 하는 조건 = 테스트 컨디션으로 볼 수 있음. 작성법 2가지:

  • 시나리오 기반: Given/When/Then (BDD 형식)
  • 규칙 기반: 입력-출력 매핑 목록·표

ATDD로 테스트 도출 K3

  1. 명세 워크숍: 팀이 스토리·인수 조건을 분석·토론·작성 (불완전성·모호성 해결)
  2. 테스트 케이스 작성: 인수 조건 기반, "소프트웨어가 어떻게 동작하는지에 대한 예제"
  • 순서: 긍정(유효) 테스트 먼저 → 부정(비유효) → 비기능
  • 이해관계자가 이해할 수 있는 표현 (자연어, 전제 조건·입력·사후 조건)
  • 스토리의 모든 특성을 다루되 스토리를 벗어나면 안 됨, 두 TC가 같은 특성을 다루면 안 됨
  • 자동화 프레임워크 형식으로 쓰면 → 인수 테스트가 "실행 가능한 요구사항"이 됨

5.1 테스트 계획

테스트 계획서 K2

목적: 방법·일정 문서화, 기준 충족 지원, 의사소통 수단, 정책·전략 준수 표명. 내용: 정황(범위·목적·베이시스), 가정/제약, 이해관계자, 의사소통, 리스크 목록, 테스트 접근법, 예산/일정.

릴리스 계획 vs 반복 주기 계획에서 테스터의 기여 K1

  • 릴리스 계획: 테스트 가능한 스토리·인수 조건 작성 지원, 리스크 분석 참여, 테스트 노력 추정, 접근법 결정
  • 반복 주기 계획: 스토리별 상세 리스크 분석, 테스트 용이성 판단, 업무 분류, 업무별 노력 추정

시작 조건 / 완료 조건 K2

  • 시작 조건: 활동 수행의 전제 (자원·테스트웨어 가용성, 초기 품질 수준 예: 스모크 테스트 통과). 애자일 = 준비의 정의(DoR)
  • 완료 조건: 종료 선언 기준 (커버리지 달성, 미해결 결함 수, 결함 밀도 + 예/아니오 기준). 애자일 = 완료의 정의(DoD)
  • 시간·예산 소진도 유효한 완료 조건 — 이해관계자가 리스크를 검토·수용했다면 종료 가능 ← 함정 선지

추정 기법 4가지 K3 — 계산 문제!

기법유형방법
비율 기반메트릭 기반과거 프로젝트의 "표준 비율" 활용. 예: 개발:테스트 = 3:2, 개발 600MD → 테스트 400MD
외삽법메트릭 기반현재 프로젝트 초기 데이터로 남은 작업 추정. 반복적 SDLC에 적합 (지난 3개 주기 평균 등)
와이드밴드 델파이전문가 기반전문가들이 독립 추정 → 편차 논의 → 재추정 → 합의까지 반복. 변형 = 플래닝 포커
3점 추정전문가 기반낙관(a)·최빈(m)·비관(b) → E=(a+4m+b)/6, SD=(b−a)/6
3점 추정 예제: a=6, m=9, b=18 → E=(6+36+18)/6=10, SD=(18−6)/6=2 → 최종 10±2 (8~12 맨아워). 공식 암기 필수!

테스트 케이스 우선순위지정 3가지 K3

  • 리스크 기반: 가장 중요한 리스크를 다루는 TC 먼저
  • 커버리지 기반: 가장 높은 커버리지 달성 TC 먼저 (변형: 추가 커버리지 우선순위 — 이후 각 TC는 가장 높은 추가 커버리지를 주는 것)
  • 요구사항 기반: 요구사항 우선순위(이해관계자가 정의) 따라
우선순위를 뒤집는 요소: 종속성(높은 TC가 낮은 TC에 종속되면 낮은 것 먼저) + 자원 가용성(도구·환경·인력)

테스트 피라미드 K1

아래층(단위) = 작고·독립적·빠름·많이 / 위층(E2E) = 느리고·큰 기능·적게. 최초 모델(Cohn): 단위 → 서비스 → UI.

테스팅 사분면 K2

브라이언 마릭 정의. 축: 비즈니스/기술 × 팀 지원/제품 평가

사분면축내용
Q1기술 + 팀 지원컴포넌트·컴포넌트 통합 테스트. 자동화 + CI 포함
Q2비즈니스 + 팀 지원기능 테스트, 스토리 테스트, UX 프로토타입, API 테스팅. 수동/자동
Q3비즈니스 + 제품 평가탐색적, 사용성, 사용자 인수 테스팅. 사용자 중심, 주로 수동
Q4기술 + 제품 평가스모크 테스트 + 비기능 테스트(사용성 제외). 주로 자동화

5.2 리스크 관리

리스크 수준 = 발생 가능성 × 영향 (가능성은 0과 1 사이). 리스크 관리 = 분석(식별+평가) + 제어(완화+모니터링). 이를 기반으로 테스트를 선택·우선순위화하는 접근법 = 리스크 기반 테스팅.

프로젝트 리스크 vs 제품 리스크 K2단골

프로젝트 리스크제품 리스크
프로젝트 관리·제어 관련: 조직 문제(지연·예산 축소), 인력 문제(기술 부족·갈등), 기술 문제(범위 추가·도구 부족), 공급업체 문제(인도 실패·파산)제품 품질 특성 관련: 기능 누락·오류, 부정확한 계산, 열악한 아키텍처, 느린 응답, 보안 취약점
영향: 일정·예산·범위 → 프로젝트 목표 달성결과: 사용자 불만족, 매출·평판 손실, 제3자 피해, 유지보수 비용, 형사 처벌, 신체 손상·사망까지

제품 리스크 분석 K2

  • 목표: 잔존 리스크 최소화 방향으로 테스트 노력 집중. SDLC 초기 시작이 이상적
  • 리스크 식별(브레인스토밍·워크숍·인터뷰) + 리스크 평가(분류, 가능성·영향·수준 결정, 우선순위, 조치 제안)
  • 평가: 정량적(가능성 × 영향 곱셈) / 정성적(리스크 행렬)
  • 결과 활용: 테스트 범위·레벨·유형 결정, 기법·커버리지 결정, 노력 추정, 우선순위지정

제품 리스크 제어 K2

완화(조치 실행으로 수준 낮춤) + 모니터링(조치 효과 확인, 새 리스크 식별). 테스팅으로 하는 완화 조치: 적절한 테스터 선정, 독립성 수준 적용, 리뷰·정적 분석, 적절한 기법·커버리지, 적절한 테스트 유형, 리그레션 포함 동적 테스팅.

5.3 테스트 모니터링, 제어, 완료

  • 모니터링 = 정보 수집 (진행 상황을 계획과 비교)
  • 제어 = 그 정보로 조치 (우선순위 재지정, 일정 조정, 자원 추가 등)
  • 완료 = 마일스톤에서 데이터·경험·테스트웨어 수집

테스트 메트릭 7분류 K1

프로젝트 진행(작업 완료율) / 테스트 진행(TC 실행·합격·불합격 수) / 제품 품질(응답 시간) / 결함(수·밀도·발견 비율) / 리스크(잔여 리스크) / 커버리지(요구사항·코드) / 비용(테스팅 비용)

진행 상황 보고서 vs 완료 보고서 K2

테스트 진행 상황 보고서테스트 완료 보고서
시기정기적(매일·매주), 비공식적일 수 있음마일스톤에서 한 번, 템플릿 따름
내용기간, 진행(빠름/늦음), 방해 요소, 메트릭, 신규 리스크, 다음 예정요약, 목적·완료조건 기반 평가, 계획과의 편차, 메트릭, 미완화 리스크·미수정 결함, 교훈

전달 방법: 대화, 대시보드(CI/CD·태스크 보드·번다운 차트), 이메일·채팅, 온라인 문서, 공식 보고서 — 대상(이해관계자)에 따라 조정. 분산 팀은 더 공식적인 방법 적합.

5.4 형상 관리 K2

  • 테스트웨어를 형상 항목으로 식별·제어·추적
  • 승인된 항목 = 베이스라인 → 공식 변경 제어 프로세스로만 수정 가능
  • 이전 베이스라인으로 되돌리면 이전 테스트 결과 재현 가능
  • 보장 사항: 모든 형상 항목에 고유 식별자 + 버전 관리 + 변경 추적 + 연관성 식별
  • CI/CD 데브옵스 파이프라인에는 보통 자동화된 형상 관리 포함

5.5 결함 관리 K3 — 결함 보고서 작성

  • 보고된 이상 사항이 다 결함은 아님 (거짓-양성, 변경 요청일 수도) → 처리 과정에서 판별
  • 워크플로우: 기록 → 분석·분류 → 대응 결정(수정/유지) → 종료. 정적 테스팅(정적 분석) 결함도 비슷하게 처리
  • 결함 보고서의 목적: ① 해결 담당자에게 충분한 정보 ② 품질 추적 수단 ③ 프로세스 개선 아이디어

결함 보고서 포함 항목 (동적 테스팅 기준)

  • 고유 식별자 / 제목(요약) / 날짜·조직·작성자(역할 포함)
  • 테스트 대상·환경 식별 정보 / 결함의 정황(실행 중이던 TC, 기법 등)
  • 장애 재현·해결에 필요한 정보 (절차, 로그, DB 덤프, 스크린샷, 녹음)
  • 기대 결과 vs 실제 결과 ← 핵심
  • 심각도(이해관계자 이익에 끼치는 영향) / 우선순위(수정 순서) — 서로 다른 개념!
  • 상태 (신규, 연기, 중복, 수정 대기, 확인 테스트 대기, 재-오픈, 완료, 거부)
  • 참조 사항 (관련 TC)
식별자·날짜·작성자·초기 상태 등은 결함 관리 도구가 자동으로 채워줌 — "전부 수동 작성" 선지는 오답

6.2 테스트 자동화의 효과와 리스크 K1

도구 도입 자체가 성공을 보장하진 않음 — 도입·유지보수·교육 노력이 항상 필요.

효과

  • 반복적 수작업 감소 → 시간 절약 (리그레션 실행, 데이터 반복 입력, 결과 비교)
  • 일관성·재현성 향상 → 사람의 단순 실수 방지
  • 객관적 평가 (커버리지 등) + 사람이 판단하기 어려운 복잡한 측정치
  • 테스팅 정보(진행률·실패율 통계·그래프)에 쉬운 접근
  • 실행 시간 단축 → 결함 조기 식별, 빠른 피드백, 출시 시간 단축
  • 테스터가 더 심층적인 새 테스트를 설계할 시간 확보

리스크

  • 도구 효과에 대한 비현실적 기대
  • 도입·스크립트 유지보수·프로세스 변경에 드는 시간·비용·노력의 부정확한 추정
  • 수동 테스팅이 더 적합한 곳에 도구 사용
  • 도구에 과의존 (비판적 사고의 필요성 무시)
  • 공급업체 종속 (폐업, 지원 중단, 매각, 열악한 지원)
  • 오픈소스 리스크 (지원 중단, 잦은 업데이트 필요)
  • 개발 플랫폼과 비호환 / 규제·안전 표준 부적합
